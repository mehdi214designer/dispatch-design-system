Stagger Rules
Stagger increment
Always 0.1s
Each sibling element in a group delays 0.1s more than the previous. Cards in a 4-column grid: 0s, 0.1s, 0.2s, 0.3s. Never use 0.05s or 0.2s increments.
nth-child(1) → delay: 0s · nth-child(2) → delay: 0.1s · nth-child(3) → delay: 0.2s
Max stagger delay
0.7s cap
Stagger delays cap at 0.7s for scroll-triggered elements. After that, the animation feels disconnected from the scroll event. Hero elements use longer delays (up to 1.4s) because they fire on page load, not scroll.
Trigger threshold
10% visible
Animations trigger when 10% of the element enters the viewport (IntersectionObserver threshold: 0.1). Once triggered, the animation plays once and the observer disconnects — no replay on scroll-out.
{ threshold: 0.1, rootMargin: '0px' }

Guidelines — Overview

How to Use This Design System

A strict, opinionated ruleset for building with the Ember design language. Every rule below is derived from the actual site — not guessed. Follow them exactly to produce consistent, on-brand results.

Core Principles
Ember's visual language is built on three ideas: warm darkness (dark backgrounds with cream text, never cold grays), sharp precision (1px radius everywhere, no softness), and restrained motion (one easing curve, one duration, always consistent). Break any of these three and the design immediately reads as off-brand.
Warm Darkness Sharp Precision Restrained Motion No border-radius between 1px and 678px No font-weight above 500 No raw hex — always use tokens
Token-first
Mandatory
Always reference CSS custom properties. Never write a raw hex color, px size, or timing value directly in component styles.
✓ background: var(--bg-elevated)   ✗ background: #262626
Warm palette only
Mandatory
All backgrounds must be warm near-black (#1c–#26). Never use cool grays (#888, slate, zinc). The warmth comes from the brown undertone in every token.
Warm: #1d1d1d #262626 #383838   Cool (forbidden): #18181b #27272a
Single accent color
Mandatory
Only one accent: #ff7038 (orange). Do not introduce secondary accent colors (blue, green, purple). Status colors (success green #8db88a, error red #c0544a) are allowed only for status indicators, never for CTAs or brand elements.
One easing curve
Mandatory
All transitions use cubic-bezier(.44, 0, .56, 1) exclusively. Never use ease, ease-in-out, linear, or any other curve for UI transitions.
transition: color 400ms cubic-bezier(.44, 0, .56, 1)
Guidelines — Tokens

Token Usage Rules

Which token to use in which context. These mappings are non-negotiable — mixing them produces off-brand results.

Color Token Mapping
--bg-primary
Page / body background
The page background. Use on body and full-width sections only. Never use on cards or interactive elements.
--bg-deep
Sidebar / below-surface
Darker than page background. Use for sidebars, deep panels, code blocks, and elements that sit "below" the main surface. Also used for the featured pricing card to create contrast without a border.
--bg-elevated
Cards / modals / inputs
The primary card background. Use on all card bodies, form inputs, dropdowns, and modal surfaces. This is the most-used background token after --bg-primary.
--surface-1
Hover state background
Only for hover states on interactive elements. Do not use as a default background. Pair with transition so the shift feels smooth.
.card:hover { background: var(--surface-1); }
--surface-2
Active / pressed state
Only for active/pressed states. One step darker than surface-1. Never use as default background.
--warm-gray
Secondary metadata
For tertiary text: timestamps, code values in token tables, separator dots. Too subtle for body copy — use --text-muted instead.
--text-primary
Headings / strong text
Headings, card titles, active nav links, button labels on dark backgrounds. Do not use for body paragraphs — that drops contrast too much.
--text-muted
Body / descriptions
The default body text color. Use for all paragraphs, card descriptions, nav links, form hints. This is the most common text token.
--text-disabled
Labels / section numbers / placeholders
For monospace uppercase labels (e.g. "01 — CONNECT"), form placeholders, and disabled element text. Never use for readable body content.
--border
All UI borders
The universal border. Apply to cards, inputs, navbars, dividers, and all structural outlines. Use at 1px solid only — never 2px or dashed.
border: 1px solid var(--border)
--border-subtle
Inner dividers / row separators
For separators inside components: between table rows, between workflow steps, between type scale rows. Lighter than --border so it doesn't compete with structural borders.
--accent
CTA / highlight / active
Primary button background, active nav links, badge foreground, focus ring, hover border, and any element that needs visual priority. Limit to 1–2 accent uses per section.
--accent-hover
Hover state of accent elements
Only use as the hover target for elements that use --accent as default. Never use as a standalone default color.
.btn-primary:hover { background: var(--accent-hover); }
--accent-dim
Badge backgrounds / tinted surfaces
12% opacity orange background. Use for badge/tag backgrounds when the badge needs to be accent-colored without being too loud. Always pair with --accent-border for the border.
Guidelines — Typography

Typography Rules

Font choice, sizing, weight, and spacing rules. Typography is where most design system violations happen — follow these precisely.

Font assignment
Mandatory
Three fonts, three purposes — never swap them:
Geist → all UI text: headings, body, nav, buttons, card text
IBM Plex Mono → all labels, badges, section numbers, code, timestamps, metrics
LT Remark (serif) → only for italic display text inside hero headings
Font weight limit
Mandatory
Only font-weight 400 and 500. Never use 600, 700, or bold. The visual hierarchy comes from size and color contrast, not weight contrast.
✓ font-weight: 400 | 500   ✗ font-weight: 600 | 700 | bold
Letter spacing
Mandatory
Headings are always tight. Scale the letter-spacing based on size:
80–72px → letter-spacing: -0.05em
56–48px → letter-spacing: -0.04em
40–32px → letter-spacing: -0.03em
24–20px → letter-spacing: -0.02em
16–14px → letter-spacing: 0 (body)
Mono labels → letter-spacing: +0.06em to +0.10em (uppercase)
Line height
Mandatory
Use em-based line heights only:
Display / hero headings → line-height: 1em
Large headings → line-height: 1.2em
Medium headings → line-height: 1.25em
Compact body → line-height: 1.35em
Standard body → line-height: 1.5em
Relaxed body → line-height: 1.6em
Mono label format
Required pattern
All monospace labels (section numbers, badges, eyebrows) must be uppercase with wide tracking. Never use sentence case for mono labels.
font-family: var(--font-mono); font-size: 10–11px; letter-spacing: .08em; text-transform: uppercase; color: var(--text-disabled);
Serif italic mixing
Display only
The serif font is only used for italic words embedded inside a large sans-serif heading. Never use the serif font for body text, buttons, or standalone headings.
<h1>Resolve Faster with <em style="font-family:var(--font-serif);font-style:italic">AI Workflows</em></h1>
Size scale discipline
Required
Only use sizes from the defined scale. Never use intermediate values like 15px, 17px, 22px, 26px. The valid sizes are:
10 · 14 · 16 · 18 · 20 · 24 · 28 · 32 · 36 · 40 · 48 · 52 · 56 · 60 · 72 · 80px
Guidelines — Spacing

Spacing Rules

Spacing must always come from the defined scale. Random px values break visual rhythm immediately.

Scale only
Mandatory
All padding, margin, and gap values must be from the spacing scale. Never use values like 5px, 7px, 9px, 11px, 13px, 15px, 17px, 19px, 21px.
Valid: 4 · 8 · 12 · 16 · 20 · 24 · 28 · 32 · 36 · 40 · 60 · 80px
Section padding
Layout rule
Top-level page sections always use 80px padding on desktop, 60px on tablet, 40px on mobile. Never reduce section padding below 40px.
Desktop: padding: 80px · Tablet: 60px 40px · Mobile: 40px 20px
Card inner padding
Component rule
Standard cards: 24px. Feature/hero cards with more content: 24px top + 32–40px bottom. Compact cards (workflow steps, badges): 12–16px.
Nav spacing
Fixed values
Navigation always: height 80px, horizontal padding 40px, link gap 32–40px. These values are fixed — do not adjust.
Grid gap pattern
Required technique
For card grids, use 1px gap on a border-colored background container instead of individual borders. This creates seamless grid lines without double borders.
.grid { display: grid; gap: 1px; background: var(--border); }
.grid-item { background: var(--bg-elevated); }
Inline element gap
Pattern
Gap between inline elements (button + icon, badge + dot): 6–10px. Gap between sibling blocks (cards in a row, buttons): 12–16px. Never exceed 24px for inline gaps.
Guidelines — Components

Component Usage Rules

When and how to use each component. Misusing a component variant is the most common design system error.

Buttons
btn-primary
One per section max
The orange CTA. Use for the single most important action per section (Get Started, Submit, Continue). Never use more than one primary button per visual section. Text on primary button must always be dark (#1d1d1d), never white.
btn-secondary
Supporting action
For secondary actions paired with a primary button (e.g. "Learn More" next to "Get Started"). Also the default button in non-CTA contexts. Uses --bg-elevated background with --border.
btn-ghost
Nav / tertiary
For navigation sign-in links, "View Docs" type actions. No background or border. Never use ghost as a primary action — it has too little visual weight.
Border radius
Always 678px
All buttons are fully pill-shaped (border-radius: 678px). Never use 4px, 8px, or any other radius on a button in this design system.
Cards
Background
Always --bg-elevated
All cards use --bg-elevated (#262626) as background. Nested elements inside cards can use --bg-deep for contrast. Never use --bg-primary as a card background — it disappears into the page.
Border
Always 1px --border
All cards have a 1px border using --border. Never use box-shadow instead of a border. Never use 2px border. Never use border-radius other than 1px.
Card label
Required pattern
Each card should have a monospace label above the title (e.g. "01 — CONNECT", "SOURCE CONNECT"). 10–11px, uppercase, --text-disabled, letter-spacing .08em.
Multi-card grids
Grid-line technique
When placing multiple cards in a grid, use the 1px gap + border background technique to create seamless grid lines. Never add individual borders to each card in a grid.
.grid { display:grid; gap:1px; background:var(--border); }
Forms
Input border radius
Always 1px
All form inputs, textareas, and selects use border-radius: 1px (sharp). Never use rounded inputs in this design system — that conflicts with the sharp, precise aesthetic.
Focus state
Mandatory
Focus state must change border-color to --accent (#ff7038). Remove the default browser outline (outline: none). The orange border IS the focus indicator.
input:focus { outline: none; border-color: var(--accent); }
Label style
Always mono
Form labels use --font-mono at 11px with letter-spacing .04em and --text-muted color. Never use sans-serif for form labels in this system.
Badges & Tags
Always pill-shaped
border-radius: 678px
All badges use the same 678px radius as buttons. Font is always IBM Plex Mono, always uppercase, always 10–11px. Never use sans-serif for badges.
Variant selection
By context
badge-accent (orange) → active status, highlighted feature, primary section labels
badge-default (bordered) → neutral tags, navigation markers
badge-warm → secondary info, muted categories
badge-mono → section numbers (001, 002), code identifiers
Navigation
Fixed dimensions
Never change
Nav height is always 80px. Horizontal padding is always 40px. Link gap is always 32–40px. These are structural constants, not suggestions.
CTA placement
Right-aligned only
CTA buttons in nav always sit on the right (flex-end). Ghost button (Sign In) precedes Primary button (Get Started). Never put a primary button in the center of the nav.
Guidelines — Layout

Layout Rules

Container widths, breakpoints, and grid structure. These define the skeleton every component sits inside.

Breakpoints
Three tiers only
Only three breakpoints. No intermediate breakpoints allowed:
Mobile → max-width: 809px
Tablet → 810px – 1279px
Desktop → min-width: 1280px
Container widths
Two values only
Outer layout: max-width 1600px. Inner content: max-width 1228px. Do not introduce other container widths. Some sections (hero, full-bleed) break out to 100% intentionally.
Grid columns
2-col or 3-col only
Feature grids are either 2-column (major features) or 3-column (blog cards, pricing). Never use 4-column or 5-column layouts. On tablet: 2-col collapses to 1-col or stays 2-col. On mobile: always 1-col.
Full-bleed sections
No horizontal padding
The ticker/marquee and some background sections are full-bleed (no horizontal padding, width: 100%). All other sections have 80px horizontal padding on desktop.
No box-shadow
Forbidden
Ember uses zero box-shadow on any element. Depth and separation come exclusively from background color steps and 1px borders. Never add drop-shadows, inner-shadows, or elevation shadows.
✗ box-shadow: 0 4px 12px rgba(0,0,0,.3)   ✓ border: 1px solid var(--border)
No gradients on backgrounds
Mostly forbidden
Section backgrounds are always flat solid colors — no gradients. Gradients are only used in overlays on top of images (e.g. screenshot caption fade) and on the CSS noise texture pattern.
Guidelines — Motion

Motion Rules

One curve, three durations, a small set of animated properties. Keep motion predictable and invisible.

The one easing curve
Mandatory everywhere
cubic-bezier(.44, 0, .56, 1) is the only easing curve used across the entire system. It's a symmetric ease — equal acceleration and deceleration. This gives Ember its calm, deliberate feel. Do not substitute with ease, ease-in-out, or any other curve.
--ease: cubic-bezier(.44, 0, .56, 1)
Duration tiers
Three only
150ms (fast) → micro-interactions: hover on small buttons, dot color changes
400ms (base) → standard transitions: nav links, card hover, border color, accordion icon, input focus
600ms (slow) → larger transitions: image zoom, page-level reveals
What to animate
Whitelist
Only animate these properties — anything else creates jank or looks inconsistent:
✓ color · background-color · border-color · opacity
✓ transform (translateX, scale — not layout-affecting)
✓ max-height (accordion only)
✓ filter (brightness/saturation on images only)
✗ Never animate: width · height · padding · margin · top/left/right/bottom
Marquee animation
Special case
The ticker uses linear easing (not the dispatch curve) at a long duration (28s–117s) to create a smooth infinite scroll effect. This is the only context where linear easing is permitted.
animation: marquee 40s linear infinite
Accordion icon
Rotate 45deg
The accordion icon (a + symbol) rotates exactly 45 degrees on open — turning it into an × symbol. Transition at 400ms. Do not use a separate close icon or animate opacity.
.acc-item.open .acc-icon { transform: rotate(45deg); }
Reduced motion
Accessibility
Respect prefers-reduced-motion. When the user has reduced motion set, disable all transforms and set durations to 1ms. The marquee animation should pause.
@media (prefers-reduced-motion: reduce) { *, *::before, *::after { transition-duration: 1ms !important; animation-duration: 1ms !important; } }
Guidelines — Anti-Patterns

What Not to Do

These are the most common mistakes that break the Ember design language. Any of these will immediately make a page feel off-brand.

✓ Do
✓Use CSS custom properties for every color value
✓Use border-radius: 1px on cards, inputs, and icons
✓Use border-radius: 678px on buttons and badges
✓Use only font-weight 400 or 500
✓Use IBM Plex Mono for all labels and badges
✓Use 1px gap grid technique for multi-card layouts
✓Apply negative letter-spacing on all headings
✓Use cubic-bezier(.44,0,.56,1) for all transitions
✓Use 400ms as the default transition duration
✓Limit primary buttons to one per section
✓Use --border (1px solid) for separation instead of shadows
✓Uppercase all mono labels with wide tracking
✓Use warm dark backgrounds (#1c–#26 range)
✓Keep section padding at 80px on desktop
✓Transition border-color to --accent on input focus
✗ Don't
✗Hardcode hex values anywhere in component styles
✗Use border-radius between 2px and 677px on anything
✗Use box-shadow for depth or elevation effects
✗Use font-weight 600, 700, or bold
✗Use Geist for monospace labels or badges
✗Use individual borders on grid cards (use gap technique)
✗Use positive or zero letter-spacing on headings
✗Use ease, ease-in-out, or linear for UI transitions
✗Use transitions faster than 150ms or slower than 600ms
✗Place two primary (orange) buttons in the same section
✗Use gradients on flat section backgrounds
✗Use sentence case for monospace labels
✗Use cool grays like #18181b, slate, or zinc
✗Reduce section padding below 40px on any screen
✗Introduce a second accent color for CTAs or brand
